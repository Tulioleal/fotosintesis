## Why

The assistant can currently violate the multilingual model-generation contract when a provider failure or action path produces user-facing assistant text from deterministic fallback strings. This is visible when grounded retrieval and answerability succeed, but final answer generation fails and the assistant degrades into prewritten prose or raw evidence instead of either generating safely through a model or returning a retryable technical failure.

## What Changes

- Remove deterministic user-facing assistant prose fallbacks from assistant graph, service, action confirmation, and tool-failure paths.
- Eliminate the minimal Spanish emergency response and any equivalent deterministic `AssistantMessage.content` fallback.
- Replace pre-redacted fallback answer strings, including RAG fallback prose that embeds raw evidence, with structured model-generation inputs.
- Preserve internal fallback drafts only as model inputs, such as `allowed_facts`, `required_points`, `prohibited_points`, intent, `answer_language`, evidence, limitations, and source support.
- Require every successful user-facing assistant message to be generated by a configured model provider from structured facts, grounded evidence, or action facts.
- Allow at most one conditional model-based recovery generation for recoverable output or prompt failures, using the same structured facts, evidence, source support, limitations, and `answer_language`.
- **BREAKING**: when no configured model provider can produce a safe assistant response, `/assistant/chat` returns a retryable machine-readable error instead of an `AssistantChatResponse` with synthetic assistant prose.
- Persist the user message for total generation failure, but do not persist an assistant message when no model-generated assistant response exists.
- Update frontend assistant chat error handling so retryable assistant failures do not append an assistant message to the thread.
- Add sanitized typed technical failure metadata for provider and tool failures so retryable provider unavailability is distinguished from semantic fallback reasons.
- Correct provider failure classification so wrapped Gemini `503 UNAVAILABLE`, rate-limit, timeout, network, and service-unavailable errors are treated as transient technical failures eligible for provider fallback and circuit breaker behavior.

## Capabilities

### New Capabilities

- None.

### Modified Capabilities

- `assistant-agent`: assistant user-facing content generation, fallback rendering, action confirmations, retryable chat failure behavior, and message persistence semantics change.
- `provider-observability`: provider fallback classification and diagnostics must preserve sanitized typed failure metadata, including wrapped transient Gemini failures.

## Impact

- Affects backend assistant graph fallback generation, RAG answer synthesis failure handling, action confirmation generation, tool-failure explanation generation, and `/assistant/chat` error behavior.
- Affects provider fallback wrappers, provider exception classification, circuit breaker eligibility, sanitized metadata, diagnostics, and tests.
- Affects assistant message persistence for total model-generation failure.
- Affects frontend assistant chat error handling for retryable machine-readable failures.
- Existing clients that assume `/assistant/chat` always returns `AssistantChatResponse.message.content` must handle retryable machine-readable errors.
